# WHIT-3323 — Slug behaviour per edition type, and what needs doing

**Abbreviations:** `override` = `slug_override`, `sft` = `slug_from_title`, `col` = `editions.slug` column. On `main`, read-time shims make `edition.slug` return `override.presence || col` and `edition.slug_from_title` return the slug column. #11371 removes both shims so readers hit the columns directly.

**States in scope:** `draft`, `submitted`, `rejected`, `scheduled`, `published`, `withdrawn`, `superseded`. Excluded: `deleted`, `archived`.

**Working assumption:** any pre-pub edition will eventually publish. A `--2` on a draft, a `nil` slug on a draft, or divergent columns on a draft all become live-URL problems on publish. Nothing about data quality is "safe to defer because it's not routing".

---

## How babosa fits in (read this before the type breakdown)

`set_slug_from_title` (`identifiable.rb:33`) calls `string_for_slug.to_slug.normalize(to_ascii: true, max_length: 150)`. No language-specific transliterator is passed — babosa uses its default transliterator, which maps Latin accented characters (é→e, ñ→n, ß→ss, etc.) to ASCII and strips everything else. Crucially, this behaviour is **content-driven, not locale-driven**:

- Latin-script or already-ASCII title → readable ASCII slug, regardless of the edition's declared `primary_locale`.
- Non-Latin-script title (Perso-Arabic, CJK, Cyrillic-without-option, etc.) → babosa strips all characters, returns empty.
- When the result is empty, the fallback at `identifiable.rb:37–40` writes `sft = document_id` (integer coerced to string), and `set_slug_from_title` returns without running the duplicate-suffix check.

Concretely: a Dari-primary edition titled "This is Dari only" (ASCII) slugifies fine → `this-is-dari-only`. A Dari-primary edition titled "این فقط دری است" (Dari script) cannot be transliterated by babosa's default tables → `sft = document_id`. The declared locale doesn't change either outcome; only the title's characters do.

This distinction matters because three separate symptoms in the type breakdown below all collapse to the same root cause: `sft` ending up as `document_id` via the babosa-empty fallback. The keep-URL checkbox in `_page_address_controls.html.erb` becomes a false choice on those rows (cross-cutting 9), because regenerating `sft` produces the same `document_id` either way.

Tightening this — e.g. passing a locale-aware transliterator to babosa, or using something other than `document_id` as the fallback — is parked as card A. This doc assumes the current behaviour stays.

**Separate point about URLs on non-English editions:** the `.dr`/`.fr`/`.pt` suffix you see on preview URLs is **not** in the slug column. `Edition#public_path` (`edition.rb:408–413`) defaults `options[:locale] ||= primary_locale`, and `ApplicationRecord#append_url_options` (`application_record.rb:9–10`) appends `.{locale}` whenever that locale isn't `I18n.default_locale`. The column stores `this-is-dari-only`; the `.dr` is applied at URL-render time. No slug-column work addresses the suffix; it's standard localised-URL rendering.

## Which types allow non-English primary (reference)

The admin UI only renders the "Create a foreign language only document" checkbox (`_locale_fields.html.erb:8`) when `edition.locale_can_be_changed?` returns `true`. `Edition::Translatable#locale_can_be_changed?` defaults to `false`; only four types override it:

| Type | `locale_can_be_changed?` | Non-English primary possible via UI? |
| --- | --- | --- |
| StandardEdition | `translatable? && translations.size <= 1` | Yes (gated) |
| Consultation | `true` | Yes |
| CallForEvidence | `true` | Yes |
| DocumentCollection | `true` | Yes |
| Publication | (default `false`) | No |
| DetailedGuide | (default `false`) | No |
| Speech | (default `false`) | No |
| CaseStudy | (default `false`) | No |
| FatalityNotice | (default `false`) | No |
| StatisticalDataSet | (default `false`) | No |
| WorldwideOrganisation | (default `false`) | No |
| LandingPage | (default `false`) | No |
| CorporateInformationPage | (default `false`) | No |

`editions.primary_locale` defaults to `'en'`, `NOT NULL` (`db/schema.rb:381`). `editions_controller#clean_edition_parameters` (line 462) strips `primary_locale` from params unless `create_foreign_language_only` was submitted — which can only be submitted when the gate renders the checkbox. So any non-English-primary row in the bottom nine types is legacy/non-UI data (pre-gate code path, import, console), not something creatable today through the admin flow.

## Type breakdown

### CorporateInformationPage

`string_for_slug` returns `nil` at `corporate_information_page.rb:118`.

- **How `sft` is set:** early-exit before babosa runs, column never written by callback.
- **What `edition.slug` returns:** always `corporate_information_page_type.slug` via `HasCorporateInformationPageType` (`delegate :slug, ...`, line 4). The slug column is never read regardless of shim state. Routing is unaffected by column values.
- **Where override comes from:** not user-settable for CIPs. Any values are from #11359 backfill.
- **Cleanup needed?** Yes — for clarity. Doesn't affect live URLs, but the dead columns mislead anyone inspecting data.
- **What to clean:** NULL `slug`, `sft`, `override` for `type = 'CorporateInformationPage'` across all seven states. Add `manages_own_slug?` predicate and skip both callbacks.

### LandingPage

Uses default `string_for_slug = non_english_edition? ? nil : title(:en)`. Does not override `locale_can_be_changed?`, so the admin UI only permits English primary. Any non-English-primary row would be legacy/non-UI data.

- **How `sft` is set:** English primary → babosa runs on `title(:en)`, typically produces a readable slug because English titles are Latin-script. Non-English primary (legacy only) → early-exit before babosa runs. Not read by routing either way.
- **What `edition.slug` returns:** column value. `set_slug` writes `override.presence || sft`; override is required (`validates :slug_override, presence: true, format: /\A\/.*\z/`, `landing_page.rb:6`), so the column equals override after any normal save.
- **Where override comes from:** user-supplied absolute path, required.
- **Cleanup needed?** Yes — `slug` and `sft` are redundant, `override` is load-bearing.
- **What to clean:** NULL `slug` and `sft` for `type = 'LandingPage'` across all seven states. Keep `override`. Same `manages_own_slug? = true`. Pre-check query 3 below — if any row has a blank override, fix it before NULLing.

### StandardEdition

All subtypes: `news_story`, `case_study`, `press_release`, `world_news_story`, `government_response`, `topical_event`, … `string_for_slug = title if primary_locale.to_sym == translation.locale` at `standard_edition.rb:183`.

- **How `sft` is set:** the locale gate passes when `translation.locale == primary_locale`. For admin-created rows the gate passes regardless of primary locale, because `editions_controller#build_edition` (`editions_controller.rb:348–352`) wraps construction in `I18n.with_locale(edition_locale)` where `edition_locale = primary_locale`. So new rows across all locales reach babosa with the title in their primary locale, and the outcome depends on whether the title's characters are transliterable — see the babosa preamble. Transliterable content → readable `sft`. Non-transliterable content → babosa-empty fallback → `sft = document_id` → `slug = document_id`. Historical non-English StandardEdition rows in the pre-pub CSV sample (ids 211711, 213586, 219640, 227356, 228363 — `it`/`pt`/`es-419`/`es`/`zh-tw`) carry `slug = document_id`, written by a pre-2013-era code path we haven't traced (current code would slugify those Latin-script locales fine; the `document_id` values are legacy). **Policy:** preserve the URL by moving the value into `slug_override` and regenerating `slug_from_title` from the title (see step 7b). Published URL stays `document_id`-based; data becomes self-describing; the keep-URL checkbox predicate works correctly on subsequent new editions of those documents.
- **Where override comes from:** user's "keep current page URL" checkbox when a title changes, plus #11359 backfill for published/withdrawn rows where `e.slug != d.slug`. Post-7b: also the `document_id`-preservation migration for historical non-English rows.
- **Cleanup needed?** Yes — regression fix (blocker, scope narrower than might be assumed), step 7b override backfill, and conditional reconciliation based on query 1.
- **What to clean:** see next steps 1, 5, 6, 7b.

### Consultation, CallForEvidence, DocumentCollection

Each overrides `string_for_slug` to return `title` with no locale gate (`consultation.rb:144`, `call_for_evidence.rb:130`, `document_collection.rb:51`).

- **How `sft` is set:** title in the edition's current locale, slugified via babosa — outcome per the preamble. Transliterable titles produce readable slugs regardless of locale. Non-transliterable titles hit the babosa-empty fallback — `sft = document_id`, and any future title edit regenerates `sft` to the same `document_id`. The URL is effectively locked to `document_id` for as long as the title stays non-transliterable.
- **What `edition.slug` returns:** column value.
- **Where override comes from:** keep-URL checkbox + #11359 backfill.
- **Cleanup needed?** Conditional on query 1.
- **Known UX issue:** for rows where `sft` resolves to `document_id` via the babosa-empty fallback, the keep-URL checkbox is a false choice. See cross-cutting 9.

### Publication, DetailedGuide, Speech, CaseStudy, FatalityNotice, StatisticalDataSet, WorldwideOrganisation

Use default `string_for_slug = non_english_edition? ? nil : title(:en)` (`identifiable.rb:103`). **None of these types override `locale_can_be_changed?`**, so `Edition::Translatable`'s default of `false` applies and the admin UI does not offer non-English primary for any of them. The `_locale_fields.html.erb` partial is hidden, the `create_foreign_language_only` checkbox never renders, and `editions_controller#clean_edition_parameters` strips `primary_locale` from params when the checkbox is absent. Column default is `'en'`, `NOT NULL`. Any non-English-primary row in these types is legacy/non-UI data (pre-gate code path, imports, console).

- **How `sft` is set:** English primary (the normal case for all new rows) → babosa runs on `title(:en)` (typically Latin-script, slugifies fine). Non-English primary (legacy rows only) → early-exit before babosa runs because `non_english_edition?` is `primary_locale != :en`. Unlike StandardEdition, the controller's `I18n.with_locale` wrapper doesn't help here — the gate hard-checks English-primary, not locale alignment. These types never reach the babosa-empty fallback; non-English rows simply leave `sft` unchanged.
- **What `edition.slug` returns:** column value.
- **Where override comes from:** keep-URL checkbox + #11359 backfill.
- **Cleanup needed?** Regression fix (scope narrower than first thought — only legacy non-English-primary rows are exposed, not anything creatable today) plus conditional reconciliation based on query 1. Query 6 sizes the legacy exposure.

## Cross-cutting issues

All are publish-time URL risks unless noted.

1. **#11371 regression — `set_slug` runs on every save.** On `main`, `set_slug` has `if: :title_changed?`. On `set-accurate-slug-value`, `set_slug_from_title` keeps the title-changed gate but `set_slug` runs unconditionally and writes `slug_override.presence || slug_from_title` to `slug`. This writes `nil` whenever both `override` is blank and `sft` is blank — regardless of whether the title changed. Most exposed groups:
   - **Publication/DetailedGuide/etc. legacy non-English primary rows:** default `string_for_slug` early-exits (before babosa), `sft` stays blank, `override` typically blank → every save overwrites `slug` with `nil`. Broken `base_path` on publish. Scope is limited to legacy data — the admin UI doesn't allow non-English primary for any of these types — but if such rows exist and are edited, the save blanks the slug. Query 6 sizes this.
   - **Published/withdrawn rows where `e.slug == d.slug`:** #11359 skipped these (override stays `nil`), and #11368 didn't touch post-pub rows (`sft` stays `nil`). Any resave — including routine republishing — now blanks the slug. `StandardEdition#base_path` is `"#{prefix}/#{slug}"` with no fallback; same for Publication, DetailedGuide, Consultation, etc. Only the generic `Edition#base_path` has the `slug || document_id.to_param` fallback. This group is the dominant exposure — it's not legacy-scoped.

   StandardEdition admin-create flow is **not** exposed — the controller's `I18n.with_locale` lets the locale gate pass, so `sft` is populated on save (to either a title-based slug or, for non-transliterable titles, `document_id` via the babosa-empty fallback — either way, non-nil). Consultation / CallForEvidence / DocumentCollection admin-create (all of which do allow non-English primary) likewise populate `sft` because their `string_for_slug` has no locale gate — babosa runs on every save and either slugifies or falls back to `document_id`. Blocker on #11371 regardless, because the post-pub group is real even without the legacy group.

2. **`sft` is blank on post-pub rows.** #11368 only covered pre-pub states. After #11371 removes the shims, two external readers hit the `sft` column directly: `admin/republishing/confirm_document.html.erb:36` (republish URL) and `_page_address_controls.html.erb:15` (checkbox predicate). Blank `sft` produces `admin/republishing/document//republish` and a checkbox predicate that's always false. Needs post-pub backfill before #11371 ships.

3. **#11368 may have introduced spurious `--N` suffixes.** `set_slug` on `main` rewrote both `slug` and `sft` to title-derived values for every pre-pub edition, running the duplicate-suffix loop against the current DB state. If the scope in `where_base_path_prefix_matches` is wider than it should be, or if the compared-to edition's `slug` column held a stale title-derived value, false collisions would have introduced `--2`/`--3`. Those suffixes then sit on drafts, and become the live URL when the draft publishes. Needs the suffix audit before any remaining drafts publish.

4. **Possible divergence between `slug` and `override.presence || sft`.** #11359 used `update_columns` to set override, bypassing callbacks. On `main`, `set_slug`'s `if: :title_changed?` means the column only reconciles when the title changes. Any edition that hasn't had a title change since #11359 could carry a `slug` column that no longer matches the effective URL. On publish, `documents.slug` is set from the edition's `slug` — so the column drives the canonical URL regardless of override. Scale unknown without query 1.

5. **Three different "slug" things get conflated:**
   - `documents.slug` — canonical URL, set from edition's `slug` at first publish.
   - `editions.slug` (column) — written by `set_slug`.
   - `edition.slug` (method) — via shim today, directly the column post-#11371. For CIPs, always delegated.

6. **Locale suffix in URLs is not in the slug column.** Explained in the babosa section above. `public_path` + `append_url_options` appends `.{locale}` at render time; the stored slug doesn't include it.

7. **Non-English StandardEdition rows with `document_id` as slug.** Small sample in CSV (5 rows); true scale sized by query 7. These are live slug values that will drive the canonical URL on publish. **Policy:** move the value into `slug_override` and regenerate `slug_from_title` from title (see step 7b). Published URL unchanged; data becomes self-describing; keep-URL checkbox predicate works correctly on subsequent new editions. Not a ship blocker for #11371 once step 1 lands — `sft` will be populated post-migration and `set_slug` will write `override.presence || sft = document_id`, preserving the URL.

8. **Sanity check on legacy types.** Rows with `type IN ('NewsArticle', 'WorldNewsStory')` shouldn't exist — those classes were migrated to StandardEdition. Any remaining rows are a separate data-integrity issue — see query 5.

9. **Keep-URL checkbox is a false choice when `sft = document_id`.** Any row that landed on the babosa-empty fallback has `sft = document_id`. On a title edit of such a row, `set_slug_from_title` runs babosa again on the new title; if the new title is also non-transliterable, `sft` is regenerated to the same `document_id`. Both branches of the checkbox (`slug_override = live_edition.slug` vs `slug_override = ""`) then resolve to the same published URL. Affected groups: Consultation/CallForEvidence/DocumentCollection (no locale gate, so any non-transliterable title is exposed); StandardEdition when the locale gate passes and the title is non-transliterable. Publication/DetailedGuide/etc. can't reach this branch — they early-exit before babosa on non-English. Not a ship blocker. Cleanup in step 8: hide the checkbox when `edition.slug_from_title == edition.document_id.to_s`. That predicate captures the actual condition at render time, without trying to infer it from locale.

## Answers to open questions

**Q: What do the `Identifiable` callbacks do on the `set-accurate-slug-value` branch?**

Two `before_save` hooks. `set_slug_from_title if: :title_changed?` calls `string_for_slug.to_slug.normalize(to_ascii: true, max_length: 150)` through babosa's default transliterator, runs a duplicate-suffix check (`--2`, `--3`) against `Edition.where_base_path_prefix_matches(self).where(slug: candidate_slug)`, and writes to `slug_from_title`. Early-exit on `string_for_slug.nil?`. Babosa-empty fallback (non-transliterable title characters) writes `sft = document_id`. `set_slug` runs every save and writes `slug_override.presence || slug_from_title` to `slug`. The read-time shims (`def slug; slug_override.presence || super; end` and `def slug_from_title; self[:slug]; end`) are deleted on this branch.

**Q: How was data set by earlier migrations?**

- **#11359** (`20260409100353_copy_slug_from_documents_to_edition_slug_override.rb`): raw SQL; for editions matching `e.slug != d.slug AND e.state IN ('published','withdrawn')`, set `override = documents.slug`. Callbacks bypassed.
- **#11368** (`20260414111505_generate_slug_from_title.rb`): iterated `Edition.in_pre_publication_state`, called `edition.set_slug`, wrote `slug` + `slug_from_title` via `update_columns(touch: false)`. Because the pre-#11371 `slug_from_title` method was the read-shim `self[:slug]`, #11368 effectively copied the existing `slug` column into the new `sft` column — it did not recompute from title, and babosa never ran. No post-pub rows touched.

**Q: Do we need to backfill published/withdrawn/superseded?**

Yes. Two external readers hit `slug_from_title` directly post-#11371 (`confirm_document.html.erb:36` and `_page_address_controls.html.erb:15`), and post-pub rows have a blank `sft`. The republish UI breaks and the keep-URL checkbox predicate is wrong. Separately, if query 1 shows real divergence between `slug` and `override.presence || sft`, reconcile `slug` too.

**Q: Why do we have to touch slugs on drafts at all?**

Because drafts publish. Any quality issue on the draft — blank `sft`, wrong `slug`, spurious `--2`, `document_id`-as-`sft` from babosa-empty — becomes the live URL on first publish or propagates into `documents.slug`. #11368 addressed pre-pub `sft` population for editions that existed at migration time, but via the shim rather than by recomputing from title through babosa. Note: new Publication / DetailedGuide / Speech / etc. drafts can only be created as English-primary via the admin UI (`locale_can_be_changed?` returns `false` for those types), so the "new non-English draft with `sft = nil`" scenario applies only to legacy rows or non-UI data routes, not admin-UI creations going forward.

**Q: Are the `--2`/`--3` suffixes from #11368 a problem?**

Yes, if any exist on drafts that will publish. Whichever slug is in the `slug` column at publish time becomes `documents.slug` (for new documents) or propagates through the override mechanism (for re-publishes). Spurious `--2`s need cleaning before the affected drafts publish. Query 4 sizes the problem.

**Q: What does a Dari (or any non-English) StandardEdition actually produce when saved? The Dari-primary test produced `this-is-dari-only.dr` — how does that square with earlier claims about non-English rows producing `nil`?**

The test worked because the title was ASCII ("This is Dari only"), not because of anything locale-specific. Walk-through:

1. Controller wraps `build_edition` in `I18n.with_locale(:dr)` (`editions_controller.rb:349`), so `translation.locale == :dr == primary_locale.to_sym` — the gate at `standard_edition.rb:183` passes.
2. `string_for_slug` returns the title "This is Dari only".
3. babosa's default transliterator handles ASCII trivially → `slug` column = `this-is-dari-only`.
4. At URL render time, `public_path` + `append_url_options` append `.dr` because the locale isn't the default → display URL is `/government/news/this-is-dari-only.dr`.

The `.dr` is a display concern; the stored slug is `this-is-dari-only`. If the same edition had been titled in Dari script, step 3 would produce an empty string, the fallback would write `sft = document_id`, and the URL would be `/government/news/{document_id}.dr`. Earlier versions of this doc conflated "non-English locale" with "non-transliterable content" — only the latter triggers the babosa-empty fallback.

**Q: Why do some non-English editions end up with `document_id`-based slugs?**

Two distinct reasons:

- **Content not transliterable by babosa.** Any title in a non-Latin script falls into the babosa-empty fallback and gets `sft = document_id`. Content-driven, not locale-driven.
- **Historical pre-2013-era rows.** A handful of non-English StandardEdition rows in the sample carry `slug = document_id` for all columns, regardless of title content. The current code path (controller `I18n.with_locale` + babosa on ASCII-capable titles) would produce readable slugs for those Latin-script locales, so these values came from code that no longer runs. Policy in step 7b: preserve the URL by moving the value into `slug_override` and regenerating `sft` from title, which makes the data self-describing without changing the published URL.

**Q: Why does the keep-URL checkbox offer a choice on some editions when the URL never actually changes?**

The babosa-empty fallback writes `sft = document_id`. On a title edit, `set_slug_from_title` runs babosa again; if the new title is still non-transliterable, `sft` regenerates to the same `document_id`, and both branches of the checkbox resolve to the same URL. The JS at `edition-form.js:173–180` only knows "title input changed or not", not "would the computed slug actually differ". Cleanup in step 8 hides the checkbox when `slug_from_title == document_id.to_s`.

**Q: Why does preview break for some WNS / Publication / `--2` examples in the diagram?**

Unverified from Whitehall code. Most likely publishing-api redirect chains when `documents.slug` and the edition's current URL disagree at preview time. Needs handover to pub-api with specific edition IDs.

**Q: Why doesn't the "keep URL" checkbox work for CIPs?**

The partial renders — `_page_address_controls.html.erb` is included in `standard_edition_form`, which every CIP form uses — but CIP slug is delegated to `corporate_information_page_type.slug`. So `live_edition.slug` returns the type slug (e.g. `"about"`), `edition.slug_from_title` is `nil` (CIP's `string_for_slug` returns `nil`, never reaches babosa), the line-15 comparison is always false, the `js-keep-slug-form-group` class is never applied, and `edition-form.js:162–184` doesn't wire up the show/hide. Visually present, functionally unwired. Low-priority cleanup: hide the partial for rows where `manages_own_slug? == true`.

**Q: Does `slug_override` silently disappear on save for press-releases?**

Not from the view as written. The `hidden_field(:edition, :slug_override, value: "")` at line 19 is the standard Rails checkbox idiom. When the checkbox is ticked, its value (`live_edition.slug`) wins over the hidden field; when unticked, the `""` clears override — which is the intended behaviour for "let URL regenerate from new title". The press-release drop reported elsewhere needs a specific repro against current code; couldn't reproduce from code review.

**Q: Does the `set_slug` split in #11371 break anything else?**

Yes — two things.

1. `set_slug` runs unconditionally. For legacy Publication/DetailedGuide/etc. non-English rows with no override, each save writes `slug = nil` (`sft` stayed blank via the early-exit, so there's nothing to fall back on) — scope limited to legacy rows because the admin UI doesn't permit non-English primary for those types. For post-pub rows with no override and no `sft` (the #11359-skipped group), each save writes `slug = nil` — this is the dominant case, not legacy-scoped. Broken `base_path` on publish.
2. Shim removal exposes blank `sft` columns to external readers. Covered above.

## Queries to run before migrating

Run on a prod replica or integration. Results scope the migration work.

```sql
-- 1. Divergence between slug column and effective URL (includes pre-pub, because pre-pub publishes)
SELECT state, type, COUNT(*)
FROM editions
WHERE slug != COALESCE(NULLIF(slug_override, ''), slug_from_title)
  AND state IN ('draft','submitted','rejected','scheduled','published','withdrawn','superseded')
  AND type NOT IN ('CorporateInformationPage','LandingPage')
GROUP BY state, type
ORDER BY 3 DESC;
```

```sql
-- 2. Blank slug_from_title on post-pub rows (for the shim-removal readers)
SELECT state, type, COUNT(*)
FROM editions
WHERE (slug_from_title IS NULL OR slug_from_title = '')
  AND state IN ('published','withdrawn','superseded')
GROUP BY state, type
ORDER BY 3 DESC;
```

```sql
-- 2b. Post-pub rows exposed to the unconditional-set_slug regression
--     (blank override AND blank sft — save blanks the slug).
SELECT state, type, COUNT(*)
FROM editions
WHERE (slug_override IS NULL OR slug_override = '')
  AND (slug_from_title IS NULL OR slug_from_title = '')
  AND state IN ('published','withdrawn','superseded')
  AND type NOT IN ('CorporateInformationPage','LandingPage')
GROUP BY state, type
ORDER BY 3 DESC;
```

```sql
-- 3. LandingPage rows with missing slug_override (must be zero before NULLing slug/sft)
SELECT state, COUNT(*)
FROM editions
WHERE type = 'LandingPage'
  AND (slug_override IS NULL OR slug_override = '')
GROUP BY state;
```

```sql
-- 4. --N suffixes anywhere, with pre-pub and post-pub broken out
SELECT state, type, COUNT(*)
FROM editions
WHERE slug RLIKE '--[0-9]+$'
  AND state IN ('draft','submitted','rejected','scheduled','published','withdrawn','superseded')
GROUP BY state, type
ORDER BY 3 DESC;
```

```sql
-- 5. Legacy type sanity check (should be zero)
SELECT type, COUNT(*)
FROM editions
WHERE type IN ('NewsArticle','WorldNewsStory')
GROUP BY type;
```

```sql
-- 6. Non-English primary rows with blank slug_from_title (will break on save post-#11371
--    for types whose default string_for_slug hard-checks non_english_edition? — they
--    never reach babosa so sft is never written).
--    For Publication/DetailedGuide/Speech/CaseStudy/FatalityNotice/StatisticalDataSet/
--    WorldwideOrganisation/LandingPage/CorporateInformationPage, any row here is a
--    legacy anomaly — the current admin UI doesn't permit non-English primary for
--    those types (locale_can_be_changed? = false). Grouping by type highlights which
--    category each anomaly falls into.
SELECT state, type, primary_locale, COUNT(*)
FROM editions
WHERE primary_locale != 'en'
  AND (slug_from_title IS NULL OR slug_from_title = '')
  AND state IN ('draft','submitted','rejected','scheduled','published','withdrawn','superseded')
GROUP BY state, type, primary_locale
```

### 7b. Move `document_id`-equal slugs into `slug_override` for StandardEdition rows (from query 7)

Data migration: for each matching row, set `slug_override = slug`, regenerate `slug_from_title` via the model callback wrapped in `I18n.with_locale(edition.primary_locale)` to let the StandardEdition locale gate pass, and re-save `slug` as `override.presence || sft`. Use `update_columns(touch: false)` to skip callbacks. Published URL unchanged; data now self-describing. Idempotent — re-run finds zero rows. Shape:

```ruby
Edition.where(type: "StandardEdition")
       .where("CAST(document_id AS CHAR) = slug")
       .where("slug_override IS NULL OR slug_override = ''")
       .find_each do |edition|
  I18n.with_locale(edition.primary_locale) do
    edition.slug_override = edition.slug
    edition.send(:set_slug_from_title)  # either slugifies, or falls back to document_id
    edition.update_columns(
      slug_override:   edition.slug_override,
      slug_from_title: edition.slug_from_title,
      slug:            edition.slug_override.presence || edition.slug_from_title,
      touch:           false,
    )
  end
end
```

If the title is in a non-transliterable script, `set_slug_from_title` will land on the babosa-empty fallback and set `sft = document_id`. That's fine — `slug_override` still preserves the URL, and the post-migration row correctly reports "this row is locked to `document_id`". The query-8 cleanup in step 8 will hide the checkbox for those rows.

**Why:** preserves the existing URL while making the data self-describing, so the keep-URL checkbox predicate (`live_edition.slug == edition.slug_from_title`) is meaningful on subsequent new editions of these documents.
